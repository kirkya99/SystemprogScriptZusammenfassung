Integer-Typen mit fest definierter Länge

Wie groß ist ein unsigned long?

  • Kommt auf das System an…
    • Linux i686: 32 Bit
    • Linux x86_64: 64 Bit
  • Problem

    Normale C-Builtin-Typen nicht plattformunabhängig!

  • Abhilfe

    Datentypen mit fest definierter Breite (C99):

    • #include <stdint.h>
    • uint8_t, uint16_t, uint32_t

Byteorder

Darstellung des Wertes 256 als 16-Bit-Zahl?

  • Big Endian

    z.B. bei PowerPC, AVR32

  • Little Endian

    z.B. Bei Intel i686, x86_64

  • Problem

    Immer noch nicht plattformunabhängig

  • Abhilfe

    Zahlen immer in Network-Byteorder (Big Endian) versenden:

    #include <arpa/inet.h>

    Beim Senden: htons, thonl

    Beim Empfangen: ntohs, nthl

Übertragung von C-Strukturen

Padding

#include <stdio.h>
#include <stdint.h>

typedef struct
__attribute__ ((packed))
{
	uint8_t type;
	uint16_t length;
} Msg;

int main(void)
{
	printf("%zu\n", sizeof(Msg));
	return 0;
}
  • Problem bei printf("%zu\n", sizeof(Msg));
    • Bei mir: 4, warum?

    ⇒ Compiler fügt Füllbytes ein

  • Abhilfe mit __attribute__ ((packed))
    • Strukturen packen!

    ⇒ Größe jetzt: 3

Arrays und Zeiger in Netzwerknachrichten

  • Falsch
    #define MAX_CONTENT_LEN 512

typedef struct
__attribute__ ((packed))
{
	uint8_t type;
	uint16_t length;
	char *content;
} Msg;

    Beinhaltet nur die Adresse!

  • Richtig
    #define MAX_CONTENT_LEN 512

typedef struct
__attribute__ ((packed))
{
	uint8_t type;
	uint16_t length;
	char *content [MAX_CONTENT_LEN];
} Msg;

    Beinhaltet die Speicher für MAX_CONTENT_LEN Zeichen

Aufbau der Chat-Nachrichten

Allgemeiner Aufbau der Nachrichten

  • Header immer gleich!
  • length gibt Länge der weiteren Daten des Paketes an
  • Strings immer in UTF-8 kodiert
  • Zahlen in Network-Byteorder!

⇒ Genauer Aufbau der verschiedenen Nachrichten ist im RFC (siehe Moodle) beschrieben!

Empfangene Daten beim Empfang immer überprüfen!

Gegenseite kann böswillig sein, (Sicherheit!) deshalb:

  • Never, ever trust unchecked data!
    • Längenangabe prüfen, darf niemals größer als der Puffer sein!
    • Nachrichtentyp sinnvoll?
    • Länge passend zum Typ
    • Nicht auf Nullterminierung verlassen!
    • Berechtigung prüfen (z.B. bei Admin-Kommandos)!
